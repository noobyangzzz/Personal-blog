# 进程在new一个对象的时候，分配的是物理内存还是虚拟内存？什么时候分配？分配的完整过程怎么样 当进程new一个新的对象的时候，实际上不会马上分配物理内存，而是会分配虚拟内存。 new的底层调用malloc,mallloc会调用更靠近内核态的ptmalloc，jemalloc,tcmalloc等（但仍然属于用户态）进行分配. 接下来进入系统调用，会进行brk()或者mmap()，这取决于分配的大小。 + 对于小块内存、堆的连续扩展，通常使用 `brk()`调整堆顶指针。 + 对于大块内存（如超过 mmap 阈值，默认 128KB 或 256KB，因系统而异），通常使用 `mmap()`映射一块新的虚拟地址区域。 但此时也只是通过brk or mmap向操作系统申请分配了一块虚拟地址空间，返回一个合法的虚拟地址空间，此时相当于进程获得了一段虚拟地址空间的使用权，但此时仍然没有真正映射到物理内存。 何时访问真正的物理内存？当我对这个new的对象进行第一次读/写的时候。此时通过虚拟地址来访问该对象，MMU负责查询页表，页面里面的PTE项有标记位，里面的标记位说明该此时物理内存的一些状态，例如次缺页（表明虚拟页已映射物理页，但页不在内存中，或页表未缓存），主缺页（虚拟页尚未映射到任何物理页）等。此时PTE就告诉MMU该虚拟页并没有映射到物理页，需要触发缺页异常。 缺页异常会触发硬件中断，需要陷入到内核态来处理。此时需要这个缺页异常程序以下事情：1.从cpu寄存器中读取引起缺页的虚拟地址 2.从cpu上下文中找到是哪个进程发生了缺页，并做一些检查（例如检查虚拟地址是否合法，该虚拟地址是否落入到了某个vm_area_struct里），之后会根据pte状态决定处理方式，例如对于次缺页会从磁盘swap回内存，或刷新TLB；对于主缺页会分配一个新的物理页框，更新页表，建立映射。 最后重新执行那条指令，成功访问物理内存。 # 面试官问我，讲一下进程调度吧。 注意在这里讲的是linux 的进程调度，对于linux，进程和线程都是task_struct 以这样的层次这样回答： 一、进程的结构体task_struct简略介绍 linux里面每个进程都有一个task_struct，这个task_struct就是我们常说的进程描述符，Linux中线程和进程都是task_struct这个结构体，区别他们是否是进程还是线程，主要是看他们是否共享同一个用户虚拟空间。 另外稍作补充，区分是内核线程还是用户进程，主要是看里面的mm_struct* mm是不是指向空，指空就是内核线程，否则就是用户进程。 每个task_struct里面有着一些调度优先级的值，例如prio,static_prio,nolmal_prio,rt_prio，这几个值就代表着某个线程对应于哪个调度类，同时在该调度类里面的“分数” 二、Linux的几个调度类 Linux内核采用模块化的调度器设计，类似优先队列链表，实现多个调度类。linux里面有五个调度类,这些调度类像是一个优先级链表，从优先级从高到底分别为停机调度类，限期调度类，实时调度类，公平调度类，空闲调度类。这几个类等于是进程调度策略的一个模块化设计。每一个调度类它内在都维护了一个数据结构和有着对应的调度策略。 三、调度过程详解 当一个任务（其实对于linux来说，就是进程）的时间片用完，那么此时就需要保存当前任务的上下文，然后切换到下一个进程。对于下面要调度哪一个线程，则是按照上面所说的优先级调度类来做的。即先从停机调度类里找任务，然后到限期调度类，然后到实时调度类，然后到公平调度类，最后到空闲调度类。 对于停机调度类，不需要任务队列，使用普通的数组这些就可以，因为通常这些任务是临时的，极其短暂的。其调度逻辑是不会被调度器主动调度，而是在系统有需要的时候发起中断，强制抢占所有当前任务，它仅供内核内部使用。 对于限期调度类，其数据结构是一个优先队列（红黑树），对应的是dl_r1(deadline run queue)，它的排序依据是截止时间越早的任务优先级越高，对应着其调度逻辑就是最早截止时间优先 对于实时调度类，其数据结构也是一个优先队列(红黑树)，对应的是rt_rq(realtime run queue)，它的调度逻辑是按照rt_prio来排序，数字越大优先级越高，针对分数相同的情况，有两种策略，分别为FIFO和时间片轮转。FIFO是指如果分数相同，取先入队列的一个；时间片轮转是，当两个任务分数相同，那么就为每个任务分配一个时间片，时间片用完就进入队列尾部。 对于公平调度类，其数据结构也是一个优先队列(红黑树),对应(cfs_rq)，键值就是vruntime,cfs称为完全公平调度器，其核心思想是计算vruntime，vruntime是指虚拟进行时间，内部有一个公式进行计算。cfs会优先调度vruntime较小的任务，确保所有任务按比例分享CPU时间。调度过程为先检查cfs_rq里是否有任务，有的话就取出vruntime最小的，运行之后更新vruntime。cfs内在还有一个“公平份额”的机制，如果当任务运行时间超过公平份额，会被抢占，之后重新插入到红黑树 空闲调度类，一般是上面的调度类都没有任务才会执行里面的任务 上面所说的每个调度类的调度策略都是可以调的，在linux里面有对应的宏定义，例如FIFO就是 SCHED_FIFO # 当我们在说进程的时候，常说是资源分配的基本单位，线程是CPU调度的基本单位。但这样的简单理解显然是不够的，操作系统对这两个概念区分开，本质上是为了给“资源集合”和“执行流”拆分开来，而进程和线程就是这样一个拆分的实例。进程可以理解成资源的容器，线程是使用这些容器中这些资源来执行任务的执行者。 进程内有独立的资源集合，他有独立的**虚拟地址空间**、**独立文件系统**、独立**文件描述符表、信号处理信息**。这样设计的目的是为了提供隔离的执行环境。每个进程创建的时候，操作系统都会为其分配独立的资源集合，通过PCB记录所有资源的状态。（） 而线程它仅仅拥有执行任务所需要的最小资源，独立的线程id，程序计数器，寄存器组，栈，TCB（线程控制块）。这些是执行流的必要信息。同一进程内线程共享进程的资源。这样设计的目的是快速切换执行流且切换开销尽可能小。 二者进行对比的话，是从三个维度进行对比： ①独占资源 每个进程拥有独立的资源集合（内存、文件描述符、环境变量、信号处理器） 线程共享进程资源，线程有栈、程序计数器、tcb等 ②切换的上下文开销，创建和销毁的开销对比： 进程切换的时候，需要切换完整资源的上下文，例如文件描述符表、信号处理规则，其中切换的主要开销是在虚拟地址空间的切换上。TLB（CPU用来缓存虚拟地址和物理地址映射的高速缓存）会全部失效，因为此时TLB缓存的是上一个进程的地址映射，对新的进程会完全失效。另外还有一个重要的切换开销是，很有可能切换的两个进程的数据和指令集不同，这就会导致L1 L2 L3缓存中的大部分内容失效。新的进程需要重新从主存中加载数据和指令，这就导致了大量的缓存未命中现象，也是所说的cpu缓存污染。 线程切换的时候，由于两个线程共享进程内的地址空间，他们的切换只需要保存上一个进程的寄存器、内核栈状态，并恢复下一个线程的寄存器和内核栈状态。切换完之后能够知道,TLB和CPU缓存都是仍然有效的。 ③健壮性 进程与进程之间，资源是完全隔离的。当一个进程崩溃（例如内存越界），不会影响其他进程 线程与线程共享进程资源，一个线程崩溃（如栈溢出）会导致整个进程崩溃。 （如果展开讲栈溢出）线程自己的栈，本质上是进程虚拟地址空间中一块被明确划分为归该线程的内存区域。而栈溢出的本质是：线程执行时，每次函数调用会将函数栈帧（包含参数、返回地址、局部变量）压入栈中，栈顶指针随之向下移动。函数返回的时候，栈帧弹出，栈顶指针向上移动。栈溢出本质山就是线程的栈空间被耗尽后，栈顶指针继续向下移动，访问到了栈边界以外的内存区域。这个时候会发生“内存保护异常”（若再展开，栈顶指针向下移，要不就访问到其他线程的关键内存区域，或是进程地址空间内未分配的内存）这个时候就会陷入内核态，终止线程 那么什么时候使用多进程多线程的呢？ 多进程：注重安全、隔离性需求 例如nginx。nginx的核心定位是 高性能高可用的HTTP反向代理和负载均衡网关。其核心目标是7*24稳定运行，即使出现单个连接处理错误，也不影响整体服务。nginx的基本架构是单master进程+多worker进程，处理连接是在worker进程当中。那么为了达成高可用的目标，他就需要使用多进程，防止某个worker线程因为处理连接错误或者网络异常等问题崩溃后，整个nginx就崩溃。如果发生了崩溃，那么master会发立即拉起新的worker进程，使得达到其高可用的目标 多线程：共享数据需求、考虑执行速率 例如workflow,这是一个搜狗做的轻量级后端业务开发框架，主要目的是为了让开发者快速构建高并发后端服务。对于后端业务开发，多任务间共享资源的情况经常发生，例如数据库连接池。若使用多进程，那么就需要各种ipc来实现，但使用多线程架构，只需要设计好任务队列，考虑好线程安全就可以了。workflow的多线程架构设计就非常适合这种io密集+轻计算的后端场景 多进程和多线程如何配合使用？ 多线程和多进程的配合，实际上就是考虑场景中的资源隔离和同一资源环境中的高效并发。进程负责“资源隔离”，线程负责“高效并发” 例如在nginx架构中，早期nginx架构中仅仅是多进程架构（master+worker，worker为单线程），但随着万级并发连接的需求增加。于是在1.7.11版本引入了线程池技术，即每个worker内也是多线程的了。这样就做到了资源隔离（单worker宕机，不影响其他worker进程） 同时也做到了多核高效（worker内多线程处理io，增加了并发量） #
